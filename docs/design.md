# Coremail 企业办公产品设计规范

本文档定义 Coremail 全产品的设计判断准则，适用于邮箱、日历、通讯录、搜索、设置、权限与共享等模块。模块特有规则放在 `docs/patterns/`，具体功能规则放在 `docs/features/`。

## 1. 设计目标

Coremail 是企业办公产品，设计必须优先帮助用户快速判断、处理和继续工作。

设计优先级：

1. 效率优先。
2. 信息清晰。
3. 布局稳定。
4. 高密度但可扫读。
5. 操作路径可预期。
6. 语义准确。
7. 状态完整。
8. 权限和安全显性。

Coremail 不是营销页、低密度展示页、消费级社交产品或 AI 包装型助手。任何视觉表达都必须服务企业任务。

## 2. 全局设计判断顺序

设计任何页面、组件或交互前，必须按以下顺序判断：

1. 用户进入当前页面要完成什么任务？
2. 用户需要判断哪些对象？
3. 哪些字段决定用户能否完成判断？
4. 哪些操作必须前置？
5. 哪些状态、权限、风险必须显性？
6. 哪些信息应该进入详情或更多？
7. 哪些内容不应该占据首屏？

不要先选择卡片、分组、图标、AI 模块或装饰风格，再反推页面结构。

## 3. 布局原则

### 3.1 稳定

同一功能入口在状态切换前后必须保持位置、尺寸和视觉语义稳定。

必须保持稳定的入口包括：

- 邮件搜索。
- 日程搜索。
- 联系人搜索。
- 视图切换。
- 同步、提醒、设置。
- 回复、转发、归档、删除。
- 共享与权限。

搜索、筛选、详情、创建、提醒和权限面板都是模块内部状态，不应变成新的顶层模块。

A/B/C 分栏使用分割线调整宽度时，分割线必须提供明确的拖拽命中区和 `col-resize` 光标；拖拽只改变当前 UI 宽度，不改变模块层级、业务数据或当前选择。邮箱、日历、通讯录、设置等普通模块共享同一套 A 栏布局偏好：桌面默认宽度为 320px，展开态在 240px 到 320px 范围内，收起态为 64px。用户在任一普通模块拖拽 A 栏宽度或手动收起 / 展开 A 栏后，切换到其他普通模块时必须保持同样的宽度或收起态，避免模块切换时左右跳动。窄屏或窄桌面触发的响应式视觉收起，也必须在当前视口下跨普通模块保持一致；用户在任一模块显式展开后，再切换模块时应继续展开，直到用户再次收起或视口规则重新生效。日历 A 栏的迷你月历年月标题必须保持单行展示，即使 A 栏拖到最小宽度也不得换行。

### 3.2 邮件三分栏

邮件工作台使用 A/B/C 三栏作为桌面端默认结构：

- A 栏是邮箱导航和文件夹范围，1728px 宽屏默认 320px，展开态最小 240px、最大 320px，收起态 64px。
- B 栏是邮件列表，1728px 宽屏默认 544px，推荐范围 480px 到 600px，最小 360px；在 A/B/C 模式中最大值必须优先保护 C 栏最小阅读宽度。
- C 栏是邮件正文阅读区，1728px 宽屏默认 864px，最小 720px，可读最大宽度建议控制在 960px 到 1040px。

邮件布局存在两种桌面模式：

- `ABC`：显示 A 栏、B 栏和 C 栏，是 1600px 及以上宽屏默认模式。
- `AB`：隐藏 C 栏，B 栏占用原 B+C 空间；用于用户主动隐藏阅读区或视口不足以稳定展示 C 栏的情况。

`AB` 是纯列表处理模式，不应把窄 B 栏卡片行简单拉宽。参考成熟企业邮箱和知识库式列表的扫读方式，纯列表必须切换为多列信息结构：

- 发件人独立列，普通收件箱默认只展示人名；邮箱地址进入详情、悬停补充或跨账号搜索来源说明，不在每行重复展示。
- 主题与摘要在主列，摘要继续单行截断；主列应获得最大可用宽度，避免被状态列挤压。
- 旗标、附件、未读等进入状态列。关联日程如果已在主题前展示，则不得在状态列重复展示。纯列表模式按当前产品决策使用紧凑状态图标，必须提供 `aria-label` / `title` 文字说明；安全风险等高影响状态仍需在详情或提示区给出文字说明。
- 时间独立右对齐列。
- 发件人、主题与摘要、状态、时间列支持用户手动拖拽调整宽度；拖拽命中区不得挤占正文信息。
- 行高保持 54px 到 64px 左右，避免大卡片化。
- 纯列表模式单击邮件行即选中该邮件并切换为已读态；双击邮件行进入模块内二级邮件详情页，不使用弹窗。
- 纯列表选中邮件后，顶部操作区必须提供当前邮件的高频处理动作，例如标为已读 / 未读、标记旗标 / 取消旗标、归档、删除。动作必须出现在带有 `已选中 1 封` 等对象说明的上下文条中，不能混入搜索框、文件夹标题或刷新 / 筛选等全局列表工具。右键菜单和 hover 操作是效率补充，不能成为唯一操作入口。
- 多选进入批量上下文后，搜索框和刷新 / 筛选等列表全局工具保持原职责；批量动作必须和 `已选中 N 封邮件` 的对象说明绑定。`AB` 纯列表模式没有 C 栏，批量动作必须出现在 B 栏选择条中；`ABC` 模式下 C 栏必须切换为批量处理面板，可使用堆叠预览表达被选对象，但堆叠中的邮件必须从头部信息、收件关系、附件、正文到末尾完整展示，不能做缩略卡、骨架卡或只显示摘要；C 栏不得恢复单封邮件工具栏，避免误判操作对象。

B/C 分割线交互规则：

- 分割线命中区至少 8px，悬浮时必须显示 `col-resize` 光标。
- 从 `mousedown` 到 `mouseup` 都必须保持拖拽，即使鼠标离开分割线或窗格也不能取消。
- 拖拽期间应禁止文本选择和误点击。
- 拖拽时优先保护 C 栏 720px 最小宽度。
- B/C 分割线采用桌面级 split view 模型，拖拽过程中必须实时反馈，不等到 mouseup 才跳变。
- 从 `ABC` 拖拽 B/C 分割线时只调整 B 栏宽度，不自动切换为 `AB`。B 栏必须在最小值和最大值之间自由拖拽；最大值由 C 栏 720px 最小阅读宽度决定。
- `AB` 纯列表模式必须通过用户显式动作进入，例如 C 栏“隐藏阅读区”或窄屏响应式规则；不能仅因为 B/C 分割线拖过某个比例就暗中切换。
- 从 `AB` 向左拖拽时，B 栏低于当前 B+C 可用宽度的 64% 后恢复 `ABC`，作为边缘恢复手柄的滞回阈值，避免误触。
- 如果未达到恢复阈值，保持 `AB`；如果处于 `ABC`，B 栏只钳制到当前 min/max，不回弹到默认值。
- `AB` 模式右边缘必须保留向内覆盖的恢复手柄，命中区不得被视口裁切到不可拖拽。

A 栏手动收起后的交互规则：

- 手动收起到 64px 后，A/B 分割线仍需保留拖拽命中区。
- 展开态 A 栏向左拖拽低于 160px 时，吸附到 64px 收起态。
- 从收起态向右拖拽达到 224px 后，A 栏恢复展开态，并在 240px 到 320px 范围内继续调整宽度。
- 160px 到 224px 是滞回缓冲区：展开态进入该区间时保持展开，收起态进入该区间时保持收起，避免拖拽过程中频繁切换。
- 窄屏自动收起不是用户手动收起；普通模块可保留 A/B 分割线作为恢复入口，但必须采用同样的展开阈值，不得一拖就展开或反复跳变。

`AB` 模式必须提供明确恢复入口：

- B 栏顶部操作区展示“显示阅读区”。
- B/C 右边缘可保留分割线或恢复手柄。
- 邮件行或更多菜单可以提供“在阅读区打开”，但不能替代顶部恢复入口。

响应式规则：

- 1600px 及以上默认 `ABC`，宽屏基准为 A320/B544/C864。
- 1320px 到 1600px 保持 `ABC`，允许 A 栏在 240px 到 320px 内压缩。
- 1144px 到 1320px 默认收起 A 栏到 64px 并保持 `ABC`，但用户可以展开 A 栏查看文件夹范围；如果展开后 C 栏无法保持 720px，则切换到 `AB` 纯列表模式。
- 小于 1144px 切换为 `AB`。
- 小于 768px 使用单列列表/正文切换，不强行展示三栏。

布局偏好应持久化，字段包括 `layoutMode`、`aWidth`、`bWidth`、`cWidth`、`isACollapsed`、`isCHidden`、`updatedAt`。其中 `aWidth` 和 `isACollapsed` 是普通模块共享的 A 栏偏好；`bWidth`、`cWidth` 和阅读区显示状态仅作用于邮箱工作台。仅在 mouseup、显示阅读区、隐藏阅读区、收起/展开 A 栏、恢复默认布局等提交动作后保存，不在拖拽过程中连续写入。

### 3.3 高密度

企业工具应优先支持扫读和比较。首屏空间必须优先给可操作内容。

优先：

- 高密度列表。
- 稳定工具栏。
- 可扫读字段。
- 紧凑筛选。
- 详情中的完整字段。

避免：

- 大卡片展示普通结果。
- 卡片嵌套卡片。
- 大面积解释性模块。
- 无任务意义的插画和装饰图标。
- 只为视觉效果增加留白。

### 3.4 层级清晰

全产品遵循：

- 左侧定范围：账号、文件夹、日历、组织、模块切换。
- 顶部定任务：搜索、筛选、视图切换、同步、提醒、主要操作。
- 主体做判断：列表、详情、阅读窗格、日历画布、联系人详情。
- 浮层做当前任务：创建、编辑、权限、确认、更多信息。

不要把同一类入口分散到多个区域，也不要让操作从当前工作上下文跳到无关层级。

### 3.5 日历时间画布

日 / 周时间画布应默认定位到用户最常处理工作的时段。当前原型使用 `08:00` 作为上班开始时间，进入日历、切换日期或切换日 / 周视图后，画布应直接显示上班时段，而不是从 00:00 开始。

休息日应作为时间画布状态呈现。中国周六、周日和法定节假日整天使用低权重灰底；调休工作日恢复普通工作日底色，并可显示 `班` 文本徽标。置灰不能隐藏创建入口、日程详情或权限状态，只表达当天默认非工作时间。

## 4. 信息展示原则

### 4.1 字段显性

企业用户完成判断通常依赖字段，而不是装饰。

不同模块的关键字段必须显性：

- 邮件：发件人、主题、摘要、时间、附件、安全、已读、旗标、文件夹。
- 日历：时间、标题、组织者、参与人、地点、来源日历、会议状态、提醒、循环。
- 通讯录：姓名、部门、职位、邮箱、电话、组织路径、外部联系人状态。
- 权限：对象、角色、权限级别、可见范围、影响范围、下一步。

字段不存在时收起，不展示空值或占位噪音。

### 4.2 语义准确

不能混淆不同业务语义：

- 发件人、收件人、抄送人不是同一种关系。
- 组织者和参与人不是同一种关系。
- 账号来源和日历来源不是同一种关系。
- 颜色分类和会议状态不是同一种信息。
- 归档、删除、彻底删除、从列表移除不是同一种操作。
- 取消共享、删除对象、撤销访问不是同一种操作。

### 4.3 来源、权限和安全

多账号、多来源、多角色场景下，必须展示足够信息帮助用户判断：

- 来自哪个邮箱账号、日历、文件夹、共享来源或组织。
- 当前用户是否有权限查看、编辑、下载、转发或共享。
- 是否涉及外部人员、外部发件人、外部链接或风险附件。
- 当前操作是否影响他人。

不要隐藏企业场景中的来源、权限、状态、安全和操作后果。

## 5. 交互原则

### 5.1 操作前置

高频操作应靠近用户当前判断对象。

示例：

- 邮件列表 hover 和右键菜单提供删除、旗标、已读/未读等快速操作。
- 邮件阅读窗格提供回复、回复全部、转发、归档、删除。
- 日程详情提供入会、编辑、删除、查看关联信息。
- 通讯录详情提供发邮件、发起会议、复制联系方式。

低频操作进入更多菜单，但不能隐藏关键工作路径。

### 5.2 后果明确

危险或有影响范围的操作必须说明后果。

必须说明影响范围的操作包括：

- 删除邮件、彻底删除邮件。
- 删除日历。
- 从共享列表移除。
- 取消共享。
- 回复全部。
- 下载或打开高风险附件。
- 修改权限。

### 5.3 状态完整

不能只实现正常态。至少考虑：

- 加载中。
- 空状态。
- 错误状态。
- 权限不足。
- 安全风险。
- 数据缺失。
- 操作中。
- 操作成功或失败。

空状态应解释为什么没有内容，并提供下一步；不能变成营销介绍。

## 6. 搜索原则

搜索是查找已知或半已知对象，不是智能推荐页面。

全产品搜索规则：

- 搜索框属于顶部工具栏。
- 搜索前后位置保持一致。
- 搜索结果以列表为主。
- 普通搜索不展示字段配置。
- 高级搜索才展示可搜索字段。
- 多账号或多来源结果必须展示来源。
- 结果过多时提供筛选、分页或继续显示。

具体规则见 `docs/patterns/search.md`。

## 7. 视觉尺度

Coremail 使用固定字号阶梯：

| 字号 | 用途 |
| --- | --- |
| 12px | 标签、数量徽标、辅助说明、低权重状态 |
| 14px | 筛选器、按钮、列表 meta 信息、普通正文 |
| 16px | 列表标题、关键字段、重要操作文本 |
| 24px | 详情页标题、页面级标题 |
| 32px | 极少使用，仅用于真正一级工作台标题 |

规则：

- 不使用 13px、15px、17px 等游离字号。
- 字号变大必须对应信息层级变高。
- 列表和搜索结果优先使用 12px、14px、16px。
- 24px 和 32px 不用于普通列表主体。

## 8. 反模式

禁止用以下方式替代清晰结构：

- 用解释性大卡片替代字段结构。
- 用装饰图标替代业务字段。
- 用 AI 包装掩盖普通搜索、筛选、摘要或分类。
- 用大留白破坏信息密度。
- 用头像替代姓名、部门、邮箱等人员识别信息。
- 用颜色替代文字表达权限、状态或安全风险。
- 用视觉风格变化制造不存在的产品层级。

## 10. 设计检查清单

设计或修改前检查：

1. 是否符合当前模块的 pattern 文档？
2. 是否保留用户当前上下文？
3. 是否展示了完成判断所需字段？
4. 是否明确了来源、权限、状态和安全风险？
5. 是否避免大卡片、营销化、AI 包装和装饰性 UI？
6. 是否覆盖加载、空、错误、权限不足和安全风险状态？
7. 是否和同类页面、同类组件保持一致？
