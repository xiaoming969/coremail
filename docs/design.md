# Coremail 企业邮箱日历设计规范

本文档用于指导 Coremail 演示产品后续页面设计与交互优化。它不是完整视觉设计系统，而是产品设计决策准则：当页面出现布局、信息架构、组件取舍或文案冲突时，优先按本文判断。

## 1. 产品定位

Coremail 是企业邮箱与日历工作台，核心价值是帮助企业用户高效处理邮件、日程、共享权限和协作信息。

设计优先级：

1. 效率优先
2. 信息清晰
3. 布局稳定
4. 高密度但可扫读
5. 操作路径可预期

本产品不是：

- 营销展示页
- 智能会议助手
- 视觉炫技页面
- 卡片式信息流产品
- 以大留白和大插画为主的消费级应用

## 2. 用户与场景

目标用户是企业组织中的高频办公用户，可能面对：

- 上万级员工通讯录
- 多邮箱账号
- 多日历来源
- 共享日历、群组日历、会议室日历
- 大量循环会议
- 权限不完整的数据
- 重名人员、无头像人员、跨部门协作

用户通常不是来“探索功能”，而是来完成任务：

- 找会议
- 查材料
- 确认时间、地点、组织者、参与人
- 查看详情
- 在可入会时一键入会
- 配置或移除共享权限
- 快速判断当前状态并继续工作

## 3. 总体设计原则

### 3.1 稳定

同一个功能入口在状态切换前后应保持位置、尺寸和视觉样式稳定。

示例：

- 搜索前后搜索框不应从工具栏跳到结果页主体。
- 搜索框不应在搜索后突然变成长条，除非用户明确进入高级搜索模式。
- 同一菜单的 hover 菜单和右键菜单内容应一致。

### 3.2 高密度

企业工具应优先支持扫读和比较，不应让装饰性空间占据首屏。

建议：

- 搜索结果优先列表，不优先卡片。
- 筛选项应紧凑排列。
- 首屏尽量展示更多有效结果。
- 多余说明应收进提示、空状态或文档，不占用主流程空间。

避免：

- 大卡片堆叠普通结果
- 大面积解释性模块
- 单条结果高度过高
- 卡片嵌套卡片

### 3.3 信息显性

判断一条企业日程是否相关，至少依赖：

- 时间
- 标题
- 组织者
- 参与人
- 地点或线上会议
- 来源账号或日历

这些字段不应混为一谈，尤其：

- 组织者和参与人不是同一种关系。
- 账号来源和日历来源不是同一种关系。
- 颜色分类和会议状态不是同一种信息。

### 3.4 少解释，多结构

页面应通过信息结构帮助用户理解，而不是用大量说明文字解释系统逻辑。

可以使用：

- 短标题
- 状态标签
- 筛选摘要
- 空状态建议
- 轻量提示

避免使用：

- 大段说明
- “系统推荐”式包装
- “命中标题/正文/附件”这类普通搜索解释
- 与任务无关的统计卡片

### 3.5 场景先于结构

页面结构必须从用户场景推导，而不是从组件样式或展示欲望推导。

设计判断顺序：

1. 用户为什么进入这个页面？
2. 用户要比较哪些对象？
3. 哪些字段决定用户能否完成判断？
4. 哪些操作必须在列表层完成？
5. 哪些信息应进入详情页或更多筛选？

示例：

- 日程搜索用户通常是在找一个已有印象的会议，不是在浏览完整未来日历。
- 共享权限用户通常是在确认权限后果，不是在管理一个抽象成员系统。
- 邮件搜索用户通常是在找一封邮件或一组材料，不是在查看营销式摘要。

因此，页面不能先假设需要大卡片、推荐区或多段分组，再把用户场景套进去。

### 3.6 视觉尺度

Coremail 使用固定字号阶梯：

| 字号 | 用途 |
| --- | --- |
| 12px | 标签、数量徽标、辅助说明、低权重状态 |
| 14px | 筛选器、按钮、列表 meta 信息、普通正文 |
| 16px | 列表标题、关键字段、重要操作文本 |
| 24px | 详情页标题、页面级标题 |
| 32px | 极少使用，仅用于真正一级工作台标题 |

规则：

- 不使用 13px、15px、17px 等游离字号。
- 日程搜索结果页原则上只使用 12px、14px、16px。
- 24px 和 32px 不用于普通搜索结果页主体。
- 字号变大必须对应信息层级变高，不能只为制造视觉强调。

搜索结果页密度基线：

- 1440px 宽屏首屏至少展示 8-10 条普通结果。
- 普通结果行高度建议 64-88px。
- 含正文或附件片段的结果行高度最多 112px。
- 筛选器高度建议 36px。
- 列表操作按钮高度建议 32px 或 36px。

## 4. 页面类型规范

### 4.1 工作台页面

适用：日历周视图、日视图、邮件列表。

原则：

- 工具栏稳定。
- 主要操作在顶部或上下文菜单中。
- 主体以可操作内容为中心。
- 左侧导航用于范围选择，不抢占主任务。

### 4.2 检索结果页

适用：日程搜索结果、邮件搜索结果。

原则：

- 搜索框保持在统一工具栏位置。
- 筛选栏紧跟结果区域。
- 主体是单一结果流，不模拟日历时间轴。
- 时间分布可以作为摘要，不应强行制造多个大分组。
- 默认展示有限数量，使用“继续显示”加载更多。

### 4.3 详情页

适用：日程详情、邮件详情、共享权限详情。

原则：

- 信息完整。
- 字段分组清晰。
- 可执行操作固定且显著。
- 详情页可以比列表页更舒展。
- 日程详情页使用文档式信息行，不使用顶部标签墙或卡片矩阵承载基础字段。
- 日程详情遵循成熟日历产品的标准事件详情模式：顶部标题和主操作，下面用紧凑字段行展示时间、方式、组织者、参与者、规则、颜色分类、发送时间、响应和正文。
- 不自造“管理信息”“推荐信息”等分组名；除正文外，基础字段优先使用同一套 icon + 字段名 + 字段值结构。
- 桌面端可使用两列字段矩阵提高空间利用率；参与者和正文可跨列展示。不要用大卡片承载基础字段。
- 列表和详情页只保留用户真正需要的入会入口，不展示“主持人入口”这类内部或低频入口。
- 组织者和参与者必须分开表达；响应、颜色分类、循环规则也必须分属不同字段。

### 4.4 弹窗与抽屉

适用：权限设置、添加共享日历、移除确认。

原则：

- 只承载当前任务。
- 主按钮文案必须表达真实语义。
- 危险操作必须解释影响范围。
- 不在弹窗中引入新的复杂模型。
- 弹窗高度上限为屏幕高度的 70%，超出内容在弹窗内部滚动。
- 日程详情类浮层使用居中模态，默认宽高为屏幕的 70%。
- 日程详情浮层内的时间、地点、组织者等基础字段使用信息行，不使用大卡片。

## 5. 日程搜索结果页规范

日程搜索结果页是普通检索页，不是智能会议助手，也不是日历视图的替代品。

### 5.1 用户目标

用户搜索日程时通常要完成：

1. 找到已有印象的会议。
2. 判断这是不是目标会议。
3. 判断这场会议和自己有什么关系。
4. 确认时间、地点、组织者和参与人。
5. 查找正文或附件线索。
6. 查看详情。
7. 在可入会时一键入会。

常见场景：

- 记得标题关键词，查近期或历史会议。
- 记得组织者或参与人，确认是否是同一场会。
- 记得附件名或正文线索，找会议材料。
- 记得大概时间，查“上次那个会”或“最近一次评审会”。
- 在共享日历、群组日历、会议室日历中查来源复杂的结果。
- 面对同名会议、循环会议和资源预订，需要降低噪音。

### 5.2 页面结构

推荐结构：

1. 顶部工具栏中的搜索框
2. 筛选栏
3. 结果数量与时间分布
4. 相关日程列表
5. 加载更多

主体是单一结果流。时间、来源和类型可以作为摘要、排序、筛选或行内标签出现，但不应默认把页面切成多个大块。

不建议默认分成：

- 即将开始
- 即将到来
- 更远未来
- 以前

原因：

- 搜索是检索任务，不是浏览时间轴。
- 大时间分组会打断扫读。
- 上万级企业中远期循环会议和共享日历容易制造噪音。
- 用户更需要比较候选结果，而不是理解系统为什么推荐。

### 5.3 时间处理

时间是重要字段，但不一定是页面主体结构。

推荐：

- 在结果行中展示完整时间。
- 临近会议用标签提示，例如 `5 分钟后开始`、`今天 18:30`。
- 顶部摘要显示时间分布，例如 `未来30天 13`、`30天后 3`、`以前 14`。
- 时间筛选作为筛选项提供。
- 排序中优先考虑未来 30 天和最近历史，远期循环会和资源预订靠后。

不推荐：

- 把所有未来结果直接摊开。
- 把远期结果做成大块解释性折叠卡。
- 普通搜索默认按复杂时间段分多个大区块。

### 5.4 结果数量

默认展示前 20 条结果。

当结果较多时：

- 使用 `继续显示 20 条`
- 提供 `显示全部`
- 超过 30 条时提示用户缩小筛选
- 超过 100 条时必须鼓励用户使用时间、人员、搜索范围缩小结果

不要无限滚动，也不要一次展示所有结果。

### 5.5 单条结果字段

必显字段：

- 时间
- 标题
- 组织者

存在时展示：

- 颜色分类
- 循环标识
- 地点
- 线上会议
- 参与人
- 来源账号
- 正文片段
- 附件信息

字段缺失时不展示空值。

不要展示：

- 地点：无
- 附件：0
- 正文：无
- 未分类
- 暂无会议链接

### 5.6 列表排版

宽屏：

- 可以使用列式列表。
- 时间独立列。
- 会议标题和标签在主列。
- 组织者、参与人、地点/线上、相关内容可分列。
- 操作列靠右。
- 普通结果行高度控制在 64-88px。
- 标题使用 16px，meta 信息使用 14px，标签和数量使用 12px。

中等宽度：

- 时间独立列。
- 标题下方使用紧凑 meta 行展示组织者、参与人、地点/线上、来源。
- 正文或附件片段单独一行，但高度应克制。

窄屏：

- 单列展示。
- 字段使用短标签。
- 操作按钮不应挤压标题和时间。
- 不隐藏时间、标题、组织者这三个基础字段。

### 5.7 操作

列表页只提供：

- 查看详情
- 一键入会

一键入会只在满足以下条件时展示：

- 有有效线上会议链接
- 会议未取消
- 会议即将开始或正在进行
- 当前账号具备访问权限

## 6. 筛选规范

筛选栏应与结果强关联，不做普通说明文字。

推荐主筛选：

- 搜索范围
- 时间
- 颜色分类
- 排序

可收起的次级筛选：

- 人员
- 会议类型
- 有附件

多账户场景：

- 账号筛选放在第一位。
- 默认当前账号。
- 跨账号结果必须展示来源账号。

单账户场景：

- 不展示账号筛选。

## 7. 颜色分类规范

颜色分类是用户自定义管理维度，不是会议状态。

展示规则：

- 有分类时显示颜色点加名称。
- 默认展示一个主分类。
- 多分类可显示 `+N`。
- 分类名称允许编辑。

不展示：

- 未分类
- 颜色分类：无

## 8. 循环会议规范

循环会议必须用文字说明，不能只用图标。

推荐：

- `循环 · 每周`
- `循环 · 每月`
- `循环 · 工作日`

普通会议不展示循环标识。

## 9. 共享权限规范

共享权限相关交互必须准确表达语义。

共享日历菜单：

- 颜色
- 查看权限
- 从列表移除

移除共享日历时：

- 文案使用 `从列表移除`
- 按钮使用 `移除` 或 `从列表移除`
- 说明必须明确：只会从我的列表移除，不影响共享方日历或权限设置

不要使用：

- 删除共享日历
- 删除权限
- 取消共享

除非真实行为确实会影响共享关系。

## 10. 文案规范

文案应直接描述用户任务或系统状态。

推荐：

- `共找到 38 条结果`
- `清除筛选`
- `查看详情`
- `一键入会`
- `从列表移除`
- `部分日历因权限限制不可搜索`

避免：

- 智能推荐
- 命中来源
- AI 分析
- 搜索内容配置
- 未设置地点
- 暂无会议链接

普通搜索不展示搜索字段配置；高级搜索才展示可搜索字段。

## 11. 响应式规范

页面应在不同宽度下保持同一信息架构，而不是换成另一套产品逻辑。

原则：

- 控件位置稳定。
- 字段可折行，但关系不变。
- 列表可从表格式变为 meta 行式。
- 不因屏幕变窄而隐藏关键判断字段。
- 操作按钮在移动端可换行，但不可遮挡标题和时间。

## 12. 反模式清单

遇到以下设计应优先重审：

- 搜索页做成日历时间轴
- 普通结果使用大量大卡片
- 搜索前后搜索框位置或尺寸剧烈变化
- 组织者和参与人混在一个字段里
- 多账号搜索不展示来源账号
- 字段不存在仍展示空值
- 一次性铺开大量循环会议
- 用解释性卡片替代清晰筛选
- 同一操作在 hover 菜单和右键菜单中不一致
- 危险操作文案不说明影响范围

## 13. 设计检查清单

提交页面改动前，至少检查：

- 这个页面的主任务是什么？
- 首屏是否展示了足够多的有效信息？
- 是否有不必要的大卡片或解释模块？
- 控件在状态切换前后是否稳定？
- 字段缺失时是否收起？
- 组织者、参与人、来源账号是否区分清楚？
- 是否把筛选维度误做成页面结构？
- 操作文案是否准确表达真实语义？
- 宽屏、中屏、窄屏是否保持同一套信息架构？
